SOLID

Single Responsibility Principle (SRP)

Functionality should be related to a single responsibility (and have only one reason for changing).

Open-Close Principle (OCP)

Software entities such as modules, classes, functions, etc. should be open for extension, but closed for modification.

Liskov Substitution Principle (LSP)

We can successfully replace the object/instance of a parent class with an object/instance of the child class, without affecting the behavior of the base class instance. (If S is a subtype of T, then objects of type T should be replaced with objects of type S).

Interface Segregation Principle (ISP)

Clients should not be forced to implement any methods they do not use. Rather than one fat interface, numerous little interfaces are preferred.

Dependency Inversion Principle (DIP)

High-level modules/classes should not depend on low-level modules/classes. Instead, both should depend upon abstractions

Domain Driven Design

  • Domain: Represents the sphere of knowledge and activity around a specific business or problem area. It includes the core business logic and rules.
  • Domain Model: An abstract model that represents the key entities, their attributes, and the relationships within the domain. Should be understandable by both domain experts and developers.
  • Ubiquitous Language: A common language shared by all team members (developers, domain experts, stakeholders). Ensures clear communication and understanding of the domain concepts.
  • Entities: Objects with a distinct identity that persists over time and across different states. Examples include Customer, Order, or Product.
  • Value Objects: Objects that are defined by their attributes rather than a distinct identity. Immutable and interchangeable if they have the same attributes. Examples include Money, Date, or Address.
  • Aggregates: A cluster of entities and value objects that are treated as a single unit for data changes. Have a root entity (aggregate root) that ensures the integrity of the aggregate.
  • Repositories: Abstractions for accessing and storing aggregates. Encapsulate the logic for retrieving and persisting entities.
  • Factories: Used for creating complex objects and aggregates. Encapsulate the creation logic to ensure the correct and consistent creation of entities.
  • Services: Operations or functionalities that do not naturally fit within entities or value objects. Can be domain services (part of the domain model) or application services (part of the application layer).
  • Bounded Contexts: Explicit boundaries within which a particular domain model is defined and applicable. Helps to manage complexity by separating different models and contexts.
  • Context Map: A high-level view of how different bounded contexts relate to each other. Shows the interactions and dependencies between different parts of the system.
  • Anti-Corruption Layer (ACL): A protective layer that translates between different models or bounded contexts. Ensures that changes in one context do not corrupt another.
  • Domain Events: Represent significant occurrences or changes within the domain. Used to communicate changes and trigger actions within or across bounded contexts.
  • Event Sourcing: A pattern where state changes are captured as a sequence of events. The current state is reconstructed by replaying these events.
  • CQRS (Command Query Responsibility Segregation): A pattern that separates read and write operations. Commands change the state (write), while queries retrieve data (read).
  • Modules: Logical groupings of related concepts within a bounded context. Helps to organize the domain model and manage complexity.

OAuth 2.0

OAuth 2.0 is an authorization framework that allows third-party applications to obtain limited access to a user's resources without exposing their credentials. It provides a way for applications to interact with each other on behalf of the user without needing to directly access their password

Key Concepts
  • Resource Owner: The user who owns the data and can grant access to it.
  • Client: The application requesting access to the user's resources on their behalf.
  • Resource Server: The server hosting the protected resources. It verifies the access tokens and serves the requested resources.
  • Authorization Server: The server issuing access tokens to the client after authenticating the user and obtaining their authorization.
  • Authorization Grant: The method by which the client obtains an access token. There are several types of grants (flows).
  • Access Token: A token that the client uses to make authenticated requests on behalf of the resource owner.
  • Refresh Token: A token used to obtain a new access token without requiring the user to re-authenticate.
Authorization Flow Grants
  • Authorization Code Grant
    • Used by web and mobile applications to obtain access tokens securely via the backend server.
    • Advantages: Secure because the access token is not exposed to the user agent and is exchanged server-to-server.
  • Implicit Grant
    • Used by single-page applications (SPAs) and public clients where the client secret cannot be stored securely.
    • Advantages: Simpler to implement for clients that run entirely in the user's browser.
  • Resource Owner Password Credentials Grant
    • Used when the client is highly trusted by the resource owner, such as first-party clients.
    • Advantages: Direct and straightforward for trusted clients.
    • Disadvantages: Not recommended for third-party clients due to security concerns of exposing user credentials to the client.
  • Client Credentials Grant
    • Used for machine-to-machine (M2M) applications where no user is involved.
    • Advantages: Suitable for server-to-server communication.
